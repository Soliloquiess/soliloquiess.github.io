## 1. MCP는 언제 쓰나 — context7 예시

> MCP는 **모델의 내장 능력 밖에 있는 것**에 연결이 필요할 때 쓴다. 실시간 외부 시스템, 최신 데이터, 제3자 API/SaaS, 사내 DB 등 — 학습 데이터(모델 가중치) 안에 없는 정보나 행동이 필요한 순간이 판단 기준이다.

### 개념 한 줄
**"모델이 이미 알고 있는가, 아니면 지금 나가서 확인해야 하는가"가 MCP를 쓸지 말지의 기준이다.**

### context7이란
context7은 **라이브러리·프레임워크·SDK의 "최신 공식 문서"를 실시간으로 가져오는 MCP 서버**다. 모델의 학습 데이터는 특정 시점에 고정돼 있어 최신 버전의 API·설정을 모를 수 있는데, context7은 그때그때 공식 문서를 조회해 이 간극을 메운다.

### context7을 쓸 때 / 안 쓸 때

| 구분 | 상황 |
|------|------|
| ✅ **쓴다** | 특정 라이브러리·프레임워크·SDK·CLI의 API 문법, 설정, 버전 마이그레이션, 설치법을 물을 때 |
| ✅ | "잘 아는 라이브러리"라도 최신 변경 반영을 위해 사용 권장 — 웹 검색보다 우선 |
| ❌ **안 쓴다** | 리팩터링, 처음부터 스크립트 작성 |
| ❌ | 비즈니스 로직 디버깅, 코드 리뷰 |
| ❌ | 일반 프로그래밍 개념 설명 |

### "언제 MCP인가"의 일반 원칙

| 조건 | 판단 |
|------|------|
| 최신성이 중요 (버전·정책·시세 등 계속 바뀜) | MCP 필요 |
| 외부 시스템에 접근해야 함 (SaaS, DB, API) | MCP 필요 |
| 독점 데이터(사내 문서, 개인 워크스페이스 등)에 접근 | MCP 필요 |
| 순수 추론·이미 아는 지식만으로 해결됨 | MCP 불필요 |
| 로컬 코드 조작(리팩터링·버그 수정 등)만 필요 | MCP 불필요 |

> 한 줄 요약: **MCP는 "모델이 모르거나 닿을 수 없는 것"에 연결할 때 쓴다.** context7은 그 대표 사례로, 라이브러리 최신 공식 문서를 실시간 조회해준다 — 단, 리팩터링·로직 디버깅 같은 순수 코드 작업엔 불필요하다.

## 2. Notion 커넥터

> **커넥터(connector)**는 Claude(claude.ai 등)가 제공하는 MCP 기반 SaaS 연동 기능이다. Notion·Linear·Asana·Figma·Box·Canva·HubSpot·Intercom 등이 있고, 그중 하나가 **Notion 커넥터**다.

### 개념 한 줄
**Notion 커넥터 = Claude를 사용자의 Notion 워크스페이스에 연결해, 페이지·데이터베이스를 검색·읽기·생성/수정할 수 있게 하는 MCP 연동.**

### 인증과 도구 노출

| 항목 | 내용 |
|------|------|
| **인증 방식** | OAuth — 사용자가 Notion에 로그인하고 권한을 승인 |
| **도구 이름 형식** | `mcp__claude_ai_Notion__<도구>` — `mcp__<서버>__<도구>` 규칙과 동일 🔗 [2026-06-27 노트 §10](/2026/06/27/claude-code-notes_2.html) |
| **가능한 동작** | 페이지·DB 검색, 내용 읽기, 페이지 생성·수정 |

### 언제 쓰나

| 상황 | 활용 |
|------|------|
| 회의록·문서·위키가 Notion에 있음 | Claude가 읽어서 요약·정리 |
| 결과물을 남겨야 함 | 작업 결과를 Notion 페이지로 기록 |
| 할 일·이슈를 Notion DB로 관리 | Claude가 조회·갱신 |

### 연결하는 법 (실제 화면)

claude.ai에서 Notion 커넥터를 연결하는 과정을 화면 그대로 따라가면 다음과 같다.

1. **설정 열기** — claude.ai 좌하단 계정 메뉴에서 **설정**(단축키 Ctrl+Shift+,) 진입.

   ![claude.ai 계정 메뉴에서 설정 진입](/assets/notion-connect-1-settings.png)
   > 그림: claude.ai 좌하단 계정(이메일) 클릭 → 설정. 홈의 "일상적으로 사용하는 도구 연결하기" 항목으로도 갈 수 있다.

2. **커넥터 탭 열기** — 설정 왼쪽 메뉴에서 **커넥터** 선택. 이미 연결된 커넥터(예: GitHub 연동)가 보이고, 우상단 **추가** 버튼으로 새 커넥터를 붙인다.

   ![설정 > 커넥터 탭](/assets/notion-connect-2-connectors.png)
   > 그림: 설정 > 커넥터. 연결됨/연결되지 않음 필터와 우상단 "추가" 버튼. 여기 예시엔 GitHub 연동이 이미 "연결" 상태.

3. **커넥터 디렉터리에서 찾기** — "추가"를 누르면 **디렉터리**가 열린다. "Anthropic 및 파트너" 커넥터 목록에서 검색창에 `Notion`을 입력해 찾는다.

   ![커넥터 디렉터리](/assets/notion-connect-3-directory.png)
   > 그림: 커넥터 디렉터리. Anthropic 및 파트너가 제공하는 커넥터들(Mozilla MDN, Realize MCP, Vaisala Xweather 등)이 카드로 나열되고, 상단 검색창으로 원하는 커넥터를 찾는다.

4. **Notion 커넥터 선택** — Notion 상세 화면. 소개: *"Connect your Notion workspace to search, update, and power workflows across tools."* 제공 **도구 13개**(search·fetch·create-pages·update-page·move-pages·duplicate-page·create-database·update-database … +5), 카테고리는 "생산성". 우상단 **연결** 버튼을 누른다.

   ![Notion 커넥터 상세](/assets/notion-connect-4-notion-detail.png)
   > 그림: Notion 커넥터 상세. 제공 도구 13개(search·fetch·create-pages·update-page 등)가 나열되고, "신뢰하는 개발자의 커넥터만 사용하라"는 경고도 함께 표시된다.

   → 여기 나열된 13개 도구가 바로 §1·§3에서 말한 MCP의 **Tools**(모델이 호출하는 함수·액션)에 해당한다.

5. **OAuth 권한 승인** — "Notion MCP에 연결하기" 화면에서 **워크스페이스를 선택**하고, Claude가 갖게 될 권한(페이지·데이터베이스 권한 존중 / 권한 수준에 따라 사용자 대신 작업 수행 / 연결된 앱 전반 검색 — Notion AI 요금제 필요 / 워크스페이스 사용자·이메일 확인)을 확인한 뒤 **계속하기**를 누르면 연결 완료.

   ![Notion MCP OAuth 권한 승인](/assets/notion-connect-5-oauth.png)
   > 그림: OAuth 권한 부여 화면. 워크스페이스를 고르고 Claude에 줄 권한을 확인한다. "연결된 앱 검색"은 Notion AI 요금제가 필요하다.

### 주의할 점

| 항목 | 내용 |
|------|------|
| **권한 범위** | 연결 = 해당 워크스페이스 내용에 Claude가 접근 가능해진다는 뜻 |
| **데이터 노출** | 사내 민감 정보가 있는 워크스페이스라면 회사 정책부터 확인 |
| **OAuth 승인 범위 확인** | 위 5단계에서 고른 워크스페이스가 곧 Claude의 접근 범위 — 승인 화면의 권한 목록(페이지·DB 권한 존중, 대행 작업, 연결된 앱 전반 검색 등)을 실제로 읽고 넘길 것 |

> 한 줄 요약: **Notion 커넥터는 OAuth로 Notion 워크스페이스를 연결해 검색·읽기·쓰기를 가능케 하는 MCP 기반 SaaS 연동**이다. 문서를 Notion에 두고 요약·기록·조회를 맡기고 싶을 때 쓰되, 권한 범위와 데이터 노출은 항상 확인할 것.

## 3. MCP vs Skill — 자주 헷갈리는 차이

> 둘 다 "Claude가 더 잘 일하게 해주는 것"이라 헷갈리기 쉽지만, 하나는 **연결(무엇에 닿을 수 있나)**, 하나는 **방법(어떻게 하나)**을 다루는 서로 다른 층이다.

### 개념 한 줄
**MCP = 외부 도구·데이터에 "연결"하는 규약(새 능력), Skill = 작업을 "어떻게" 잘 하는지 담은 지식·절차 패키지(노하우).**

- **MCP**: Claude에게 새로운 능력(할 수 있는 행동·접근할 수 있는 데이터)을 준다. 관심사는 "무엇에 닿을 수 있나(reach)". 런타임에 도구를 호출한다.
- **Skill**: 폴더 + `SKILL.md`(마크다운) (+선택적 스크립트·리소스)로 구성. 관심사는 "어떻게 하나(know-how)". 관련될 때 컨텍스트에 로드된다(progressive disclosure).

> 참고: Skill도 스크립트를 번들하면 실제 동작을 수행할 수 있다. 다만 초점은 "무엇에 닿을 수 있나"가 아니라 여전히 **"어떻게 하나(방법)"**에 있다는 점이 MCP와의 차이다.

### 비유
MCP = 새 도구·열쇠·전화선을 쥐여주는 것(연결·접근). Skill = 매뉴얼·교육을 주는 것(방법·노하우).

### 핵심 비교표

| 축 | MCP | Skill |
|---|---|---|
| 본질 | 외부 도구·데이터 연결 규약(프로토콜) | 작업 수행 방법(지식·절차) 패키지 |
| 주는 것 | 새 능력(도구 호출·데이터 접근) | 노하우(지시·워크플로) |
| 형태 | 서버(원격 HTTP/로컬 프로세스) | 폴더 + SKILL.md 마크다운(+스크립트) |
| 로딩 방식 | 도구 스키마 등록 → 런타임 호출 | 관련 시 컨텍스트로 로드(progressive disclosure) |
| 실행 위치 | MCP 서버 프로세스(로컬 또는 원격)에서 실행 | Claude 컨텍스트 내 지시(스크립트는 로컬 실행) |
| 비유 | USB-C 포트·전화선(연결) | 매뉴얼·교육(방법) |
| 예시 | context7, Notion 커넥터, GitHub MCP | visual-companion, deep-research, `/commit`류 |

### 경쟁이 아니라 상호보완
Skill이 "이 작업은 이렇게 하고, 그 과정에서 이 MCP 도구를 써라"라고 MCP를 부르는 식으로 조합된다.

```
Skill (노하우: 이 작업은 이 순서로 진행)
  └─ 절차 중 필요한 단계에서
       MCP 도구 호출 (예: context7로 최신 문서 조회)
```

### 언제 무엇을 만드나

| 필요한 것 | 만들 것 |
|-----------|---------|
| 외부 시스템(SaaS·DB·API)에 연결해야 함 | MCP 서버 |
| 반복되는 작업 절차·노하우를 재사용하고 싶음 | Skill |

> 한 줄 요약: **MCP는 "무엇에 닿을 수 있나"(연결·능력), Skill은 "어떻게 하나"(방법·노하우)를 다룬다.** 둘은 경쟁 관계가 아니라, Skill이 절차 중간에 MCP 도구를 불러 쓰는 식으로 조합되는 상호보완 관계다.
