> 발표자료를 만들면 흔히 "AI가 만든 티"가 나는데, frontend-slides는 그걸 피하도록 도와주는 Claude Code 플러그인이다. 설치·사용법, 매뉴얼형 슬라이드를 만드는 프롬프트 원칙과 `visual-verdict` 품질 점검, 그리고 강의 예시 프롬프트가 너무 복붙이라 이를 슬래시 명령으로 재사용하는 법까지 정리한다.

---

## 1. frontend-slides란

> **frontend-slides**는 Claude Code 플러그인으로, 빌드 도구나 의존성 없이 **단일 HTML 파일** 안에 슬라이드덱 전체(마크업·스타일·동작)를 담아 생성해준다.

### 개념 한 줄
**"디자인 지식이 없어도, AI가 만든 티(AI-slop) 안 나는 발표자료를 단일 HTML 파일로 뽑아주는 플러그인."**

### 해결하는 문제
발표자료를 AI에게 맡기면 흔히 나오는 결과물 — 뻔한 그라디언트, 획일적인 레이아웃, 어디서 본 듯한 느낌(AI-slop) — 을 피하고, 시각적으로 구별되는 결과물을 만드는 게 목표다.

### 특징

| 항목 | 내용 |
|------|------|
| 제로 의존성 | 단일 HTML 파일 하나로 완결(인라인 CSS/JS), 별도 빌드 불필요 |
| 스타일 선택 방식 | 취향을 말로 설명하는 대신 **시안을 보고 고름** |
| PPT 변환 | 기존 PPT를 웹 슬라이드로 변환 가능 |
| 디자인 템플릿 | 34개 이상의 디자인 템플릿 보유 |
| 포맷 | 16:9 고정 비율, 접근성 고려 |
| 배포/내보내기 | Vercel 배포, Playwright 기반 PDF export |

출처: https://github.com/zarazhangrui/frontend-slides

> 한 줄 요약: **frontend-slides = 빌드 도구 없이 단일 HTML로, "AI 티 안 나는" 발표자료를 만들어주는 Claude Code 플러그인.**

## 2. 설치

> 다른 Claude Code 플러그인과 동일하게 **마켓플레이스 등록 → 설치**의 두 단계를 거친다.

### 개념 한 줄
**마켓플레이스 add로 카탈로그를 등록하고, install로 실제 플러그인을 설치한다.**

```
/plugin marketplace add https://github.com/zarazhangrui/frontend-slides
/plugin install frontend-slides@frontend-slides
```

설치 후 호출:

```
/frontend-slides:frontend-slides
```

> 참고: `marketplace add`는 "허브에 플러그인 카탈로그를 등록"하는 단계, `install`은 그 카탈로그에서 실제로 설치하는 단계다. 🔗 [MCP 노트](/2026/07/11/mcp-when-context7-notion-skill.html)에서 다룬 "규약 vs 허브" 구도와 같은 2단계 패턴.

## 3. 기본 사용법 — "show, don't tell" 워크플로

> 핵심은 스타일 취향을 말로 설명하지 않는다는 것. 대신 **시안 여러 개를 먼저 보여주고, 그중 마음에 드는 걸 고르게 한다.**

### 개념 한 줄
**"말로 설명(tell)" 대신 "보고 고르기(show)"로 스타일을 결정하는 워크플로.**

### 진행 흐름

| 단계 | 내용 |
|------|------|
| 1 | 발표 내용 수집 |
| 2 | 스타일 시안 3개 생성 |
| 3 | 사용자가 마음에 드는 시안 선택 |
| 4 | 선택한 스타일로 최종 슬라이드 생성 |
| 5 | 브라우저에서 결과물 오픈 |

기존 PPT가 있는 경우, 텍스트·이미지·레이아웃을 유지한 채 웹 슬라이드로 변환도 가능하다.

<!-- [스크린샷 자리] 파일 예: /assets/frontend-slides-5-style-c.png — 스타일 시안 3 (선택 화면) -->

> 한 줄 요약: **스타일을 말로 설명하지 말고, 시안 3개 중 골라라.** 발표 내용 수집 → 시안 3개 → 선택 → 최종 생성 → 브라우저 확인의 5단계면 충분하다.

## 4. 강의에서 강조한 "매뉴얼형" 제작 원칙

> 이건 플러그인이 자동으로 해주는 기본 기능이 아니라, **사용자가 프롬프트로 직접 지시해야 하는 제작 규칙**이다. 특히 화면 캡처가 들어가는 사용 설명서·매뉴얼형 슬라이드를 만들 때 유용하다.

### 개념 한 줄
**"슬라이드당 한 메시지 + 바운딩 박스 + 이미지 재활용 + spotlight 강조" — 네 가지를 프롬프트에 명시해야 매뉴얼답게 나온다.**

### 원칙 정리

| 원칙 | 내용 |
|------|------|
| 슬라이드당 한 메시지 | 한 슬라이드엔 설명 하나만 담는다. 설명할 게 여럿이면 슬라이드를 나눈다 |
| 바운딩 박스 | 캡처 위에 박스를 그려 "어디를 클릭하면 화면이 어떻게 바뀌는지" 디테일하게 표시 |
| 이미지 복제 재활용 | 한 캡처에서 설명할 포인트가 여럿이면, 같은 이미지를 복제해 슬라이드마다 다른 부분을 강조 |
| spotlight 강조 | 특정 위치를 스포트라이트처럼 부각시켜 시선을 유도 |

<!-- [스크린샷 자리] 파일 예: /assets/frontend-slides-bbox.png — 캡처 위에 바운딩박스 + spotlight를 적용한 매뉴얼 슬라이드 예시 -->

### 왜 이게 "AI스럽지 않은" 결과를 만드나
AI가 기본값으로 만드는 발표자료는 한 슬라이드에 정보를 욱여넣거나, 캡처를 그냥 붙여넣기만 하는 경우가 많다. 위 네 원칙은 실제 사람이 만드는 사용 설명서의 관례(한 동작 = 한 슬라이드, 클릭 지점 명시, 반복 캡처 재활용)를 그대로 프롬프트로 강제하는 것이라, 결과물이 "누가 직접 만든 매뉴얼"처럼 보이게 된다.

> 한 줄 요약: **매뉴얼형 슬라이드의 핵심은 플러그인이 아니라 프롬프트 규칙에 있다.** 슬라이드당 한 메시지·바운딩 박스·이미지 재활용·spotlight, 이 네 가지를 지시해야 AI 티가 빠진다.

## 5. 품질 점검 — visual-verdict

> 슬라이드를 다 만든 뒤에는 `/visual-verdict` 스킬로 **글자 겹침·대비(가독성) 문제**를 점검한다.

### 개념 한 줄
**생성된 슬라이드에서 텍스트가 요소와 겹치거나, 배경-글자 대비가 낮아 안 보이는 흔한 결함을 잡아내는 점검 스킬.**

> 참고: `visual-verdict`는 frontend-slides에 포함된 기능이 아니라 **독립된 스킬**이다. 그래서 플러그인 네임스페이스 없이 `/visual-verdict`로 부른다(§2의 `/frontend-slides:...` 호출과 다른 점).

| 항목 | 내용 |
|------|------|
| 점검 대상 | 글자·요소 겹침, 배경-텍스트 대비(가독성) |
| 실행 시점 | frontend-slides로 슬라이드 완성 후 |
| 결과 | 문제 위치를 짚어주고, 이어서 수정 지시 가능 |

<!-- [스크린샷 자리] 파일 예: /assets/frontend-slides-7-visual-verdict.png — visual-verdict 점검 결과 화면 -->

생성 → 점검 → 개선의 루프로 이어진다.

```
frontend-slides (생성) → /visual-verdict (점검) → 수정 지시 (개선) → 재점검 …
```

> 한 줄 요약: **frontend-slides로 만들고 끝이 아니라 `/visual-verdict`로 대비·겹침을 점검하는 루프를 거쳐야 완성도가 올라간다.**

## 6. 반복 프롬프트를 슬래시 명령으로 (복붙 줄이기)

> 강의 예시 프롬프트는 §4의 "캡처+바운딩박스 / 슬라이드당 한 설명 / 이미지 복제 / spotlight" 블록을 매번 통째로 반복한다. 이건 재사용 가능한 **제작 규칙**이므로, 한 번만 정의해두고 계속 불러 쓰면 된다. 실제로 매번 달라지는 건 ① 소재(현재 프로젝트인지, `docs/*.pdf`인지)와 ② 분량뿐이다.

### 개념 한 줄
**매번 바뀌지 않는 규칙은 프롬프트에서 분리해, 명령(또는 파일)으로 한 번만 정의하고 재사용한다.**

### 세 가지 방법

| 방법 | 내용 |
|------|------|
| A. 자연어로 의도만 | 규칙을 짧은 문장으로 압축해 매번 간단히 전달 |
| B. 규칙 파일 참조 | `docs/slide-rules.md`에 규칙을 한 번 적어두고 "규칙은 그 파일 따라"라고만 지시 |
| C. 커스텀 슬래시 명령 (가장 깔끔) | `.claude/commands/slides.md`에 규칙을 박아두고 `/slides <소재>`로 호출 |

### C안 예시 — `.claude/commands/slides.md`

```markdown
---
description: frontend-slides로 슬라이드 생성 (우리 규칙 적용)
---
/frontend-slides:frontend-slides 스킬로 $ARGUMENTS 를 슬라이드로 만들어줘.
규칙:
- 슬라이드당 설명 하나만
- 캡처에 바운딩박스로 "클릭 지점 → 화면 전환"을 디테일하게
- 한 캡처에 설명 여럿이면 이미지 복제 재활용
- 강조는 spotlight
- 완료 후 /visual-verdict 로 글자 겹침·대비 점검
```

이후엔 아래처럼 짧게만 호출하면 된다. 규칙은 명령에 이미 박혀 있으니 두 번 쓸 필요가 없다.

```
/slides docs/*.pdf 를 50장으로 정리
```

> 참고: 커맨드 파일(`slides.md`) 본문은 실행 시 그대로 **프롬프트로 주입**된다 — 규칙 블록이 매번 자동으로 붙는 셈이라 복붙이 사라진다.

> 한 줄 요약: **복붙을 느꼈다는 건 "프롬프트를 스킬/명령으로 승격할 때"라는 신호다.** 이번 주 주제인 "프롬프트 → 하네스/명령"의 흐름과 그대로 이어진다.
