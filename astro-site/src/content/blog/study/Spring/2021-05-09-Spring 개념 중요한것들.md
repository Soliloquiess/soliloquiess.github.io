## 1. 스프링 특징과 핵심 용어

### @Transactional

트랜잭션 처리를 지원하는 어노테이션. 클래스 또는 메서드 위에 선언하는 **선언적 트랜잭션** 방식이 일반적이다.

`@Transactional`이 추가되면 해당 클래스에 **트랜잭션 기능이 적용된 프록시 객체**가 생성된다. MyBatis, JPA 모두 이 어노테이션 하나로 트랜잭션을 처리한다.

---

### IoC (Inversion of Control, 제어의 역전)

메서드나 객체의 호출 작업을 **개발자가 결정하지 않고 외부(스프링 컨테이너)에서 결정**하는 것을 의미한다.

---

### DI (Dependency Injection, 의존성 주입)

IoC가 일어날 때 스프링이 내부 객체들 간의 관계를 관리하는 기법.

- 의존 객체를 직접 생성하거나 제어하지 않고, **외부에서 결정해 연결**시킨다.
- 자바에서는 일반적으로 **인터페이스**를 이용해 유연하게 처리한다.
- 결과적으로 **모듈 간 결합도가 낮아지고 유연성이 높아진다.**

---

### PSA (Portable Service Abstraction, 이식 가능한 서비스 추상화)

인터페이스를 통한 약한 결합도와 DI를 활용해, 스프링은 변화하는 환경에서도 **일관성 있는 추상 API 계층**을 제공한다. 변경에 유연하게 대처할 수 있는 서비스 구조다.

---

## 2. AOP — Advice 용어 정리

**AOP**(Aspect-Oriented Programming)에서 **Advice**란, Target 클래스의 JoinPoint에 삽입되어 실행되는 공통 관심 로직 코드다.

> 예: "메서드를 호출하기 전(언제)에 트랜잭션을 시작한다(공통기능)"

Advice를 어디에 적용할지는 **PointCut** 단위로 정의한다.

| Advice 타입 | 실행 시점 |
|-------------|-----------|
| **Around Advice** | JoinPoint 앞과 뒤 모두 |
| **Before Advice** | JoinPoint 앞 |
| **After Returning Advice** | 메서드가 정상 종료된 뒤 |
| **After Throwing Advice** | 예외가 던져질 때 |
| **Introduction** | 클래스에 인터페이스와 구현을 추가하는 특수 Advice |

## 3. 스프링 MVC 관련 용어 5개 + 흐름

### MVC 패턴

| 구성 요소 | 역할 |
|-----------|------|
| **M (Model)** | Data / DB를 직접 다루는 부분 |
| **V (View)** | 화면에 보이는 부분 |
| **C (Controller)** | 이벤트와 제어 기능, 실질적인 처리 담당 |

### 핵심 객체

- **DAO (Data Access Object)** — CRUD를 전문적으로 다루는 클래스 (연결지향)
- **DTO (Data Transfer Object)** — DAO의 데이터를 묶어서 BL/PL 계층에 전달. 데이터 전송을 위해 캡슐화한다.

### Forward vs Redirect

| 구분 | 처리 위치 | URL 변경 | Request/Response 공유 |
|------|-----------|----------|-----------------------|
| **Forward** | 서버 내부 | 변경 없음 | 공유 (동일 객체) |
| **Redirect** | 클라이언트 재요청 | 변경됨 | 새로 생성 |

- **Redirect**: 서버가 클라이언트에 HTTP 302를 전송 → 클라이언트가 Location 헤더의 URL로 재요청
- **Forward**: 서블릿 컨테이너가 요청을 그대로 타겟 URL로 포워딩. 브라우저 URL은 최초 요청 URL 그대로

## 5. 스프링 부트 특징

1. 작업 결과를 **jar**로 패키징하여 단독 실행이 가능하다.

4. 기본 View 형식은 JSP로, 추가 dependency 없이 사용 가능하다.

## 8. 스프링 주요 어노테이션

`@Controller`, `@Service`, `@Repository`는 모두 Bean 객체를 생성하지만, **역할을 명시적으로 구분**하기 위해 분리해서 사용한다. 부모 어노테이션인 `@Component`를 써도 동작하지만 가시성이 낮아 잘 사용하지 않는다.

| 어노테이션 | 레이어 | 역할 |
|------------|--------|------|
| **`@Controller`** | 프레젠테이션 | 웹 요청·응답 처리, View 반환 |
| **`@Service`** | 서비스 | 내부 자바 비즈니스 로직 처리 |
| **`@Repository`** | 퍼시스턴스 | DB·파일 등 외부 I/O 처리 |

## 9. 동기 요청 파라미터 — @RequestParam

`@RequestParam`은 `HttpServletRequest` 객체와 같은 역할로, 쿼리스트링이나 폼 데이터를 받는다.

    @Controller
    public String hello(@RequestParam String num, @RequestParam String name){}

---

## 10. 비동기(Ajax) 파라미터 — @PathVariable

`@PathVariable`은 URI 경로상의 변수값을 받아온다.

    @GetMapping("/bars/{barId}")
    fun getBar(@PathVariable(value = "barId") barId: Optional<String>)

    @GetMapping(value="/customers/{num}")
    public String hello(@PathVariable String num){}

> URI 경로상의 변수값을 받아오는 어노테이션 → **`@PathVariable`**

---

## 11. @RequestBody / @ResponseBody 구분

| 어노테이션 | 방향 | 역할 |
|------------|------|------|
| **`@RequestBody`** | 요청 → Java 객체 | HTTP 요청 body를 자바 객체로 매핑 (JSON → Object) |
| **`@ResponseBody`** | Java 객체 → 응답 | 자바 객체를 HTTP 응답 body로 변환 (Object → JSON) |

    public String hello(@RequestBody Customer c){}
    // @RequestBody : JSON 타입 데이터를 자바 객체로 받을 때
    // @ResponseBody : 자바 객체를 JSON 형식으로 보낼 때

## 12. HTTP 상태 코드 (200~500)

| 코드 | 이름 | 설명 |
|------|------|------|
| **200** | OK | 성공적으로 처리 |
| **401** | Unauthorized | 인증 없이 인증 필요 리소스에 접근 |
| **402** | Payment Required | 예약된 코드 (현재 표준 없음, 실무에서는 403 사용) |
| **404** | Not Found | 리소스 없음 |
| **405** | Method Not Allowed | 허용되지 않은 HTTP 메서드 사용 |
| **500** | Internal Server Error | 서버 내부 오류 (설정·권한 문제, 프로그램 비정상 종료 등) |

## 13. @SpringBootApplication 구성

`main()` 클래스의 `@SpringBootApplication`은 아래 3개 어노테이션을 포함한다 (`@MapperScan`은 포함되지 않음).

| 포함 어노테이션 | 역할 |
|----------------|------|
| **`@Configuration`** | 설정 클래스 선언. Bean을 **싱글톤**으로 한 번만 생성 |
| **`@EnableAutoConfiguration`** | 자동 설정 활성화 |
| **`@ComponentScan`** | 지정 경로 이하의 `@Component`, `@Configuration` 클래스를 스캔해 Bean 등록 |

> 출처: https://sieunlim.tistory.com/10

`@Component`는 Bean을 `new`로 생성하듯이 만들며, `@ComponentScan`은 XML의 `<context:component-scan>`을 자바로 대체한다.

## 17. Spring MVC 요청 처리 순서

① 브라우저에서 요청(Request) 발생  
② **DispatcherServlet**이 단일 프런트 컨트롤러로 요청 수신  
③ **HandlerMapping**에 처리할 컨트롤러 조회 요청  
④ 적절한 컨트롤러로 요청 전달 → 비즈니스 로직 처리 후 **Model** 생성  
⑤ 컨트롤러가 모델 + 논리적 뷰 이름을 DispatcherServlet에 반환  
⑥ **ViewResolver**가 논리적 뷰 이름을 실제 View(JSP 등)로 매핑  
⑦ View로 모델 데이터 전달 → 렌더링 후 **Response** 반환

## 18. REST 서비스란?

**REST**(Representational State Transfer) — HTTP 프로토콜을 이용해 자원을 주고받는 아키텍처 스타일.

자원(이미지, 동영상, DB 데이터 등)에 고유한 **URI**(Uniform Resource Identifier)를 부여해 클라이언트와 서버 간 통신을 구조화한다.

### REST 3대 구성 요소

| 구성 요소 | 설명 |
|-----------|------|
| **Resource** | 유일한 ID(URI)를 가지는 서버의 자원. 명사형 단어로 표현 |
| **Method** | 자원을 조작하는 동사 (GET, POST, PUT, DELETE) |
| **Representation** | 서버가 응답으로 보내는 자원의 상태 표현 (JSON, XML, text 등) |

### URI 설계 원칙

- `/groups`, `/users`처럼 **직관적인 명사**로 구성
- `/groups/{groupId}/member/101`처럼 **계층적 구조** 유지
- 상위 path는 하위 path의 집합을 의미하는 단어로 구성
- CRUD 동작은 **URI가 아닌 HTTP 메서드**로 표현

## 19. Interceptor vs Filter 차이

| 구분 | 처리 시점 |
|------|-----------|
| **Filter** | 클라이언트 요청이 들어오면 가장 먼저 처리 (예: 한글 인코딩 필터) |
| **Interceptor** | Filter 통과 후 DispatcherServlet 통과 이후 처리 |

**요청 처리 순서:**  
클라이언트 요청 → **Filter** → **DispatcherServlet** → **Interceptor** → **Controller**

## 20. 스프링 Bean 스코프 — Singleton vs Prototype

스프링 컨테이너가 객체를 생성하면 기본이 **singleton** 방식이다. 매번 새 객체가 필요하다면 `scope="prototype"`을 지정한다.

`<bean id="c" class="com.test.Car" scope="prototype"/>`
