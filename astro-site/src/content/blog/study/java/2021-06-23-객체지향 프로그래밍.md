## 기능 위주 vs 데이터 위주로 묶기

![기능 위주 vs 데이터 위주 비교](/assets/20210623_171558.png)

→ **데이터 위주로 묶어야 한다.**

- 기능 위주로 묶으면 함수의 독립성이 깨진다.
- 함수는 외부의 수정을 받으면 안 된다.
- 함수가 아무런 수정도 하지 않았는데 바뀌면 안 되므로, 그것을 방지하기 위해 매개변수까지 넣었는데도 바뀌는 경우를 방지하기 위함이다.

이런 식으로 리스트가 처리되고 사용된다.

### 인스턴스 메서드 구현하기 — 객체지향적인 함수 호출로 변경하기

- `static`은 고전적인 함수다. 모든 값을 파라미터로 넘겨받고 일반적인 함수와 똑같이 사용한다.
- 인스턴스 함수는 반드시 객체를 통해서 쓰고, 객체를 묵시적으로 넘겨받는다.
- `static`을 붙여서 정의하면 일반 함수, `static`을 빼고 구현하면 인스턴스 메서드가 된다.

똑같은 로직인데 파라미터로 전달받은 것이 없다. 그럼 어떻게 쓸 수 있을까? 인스턴스 메서드는 이를 위해 `this`라는 예약어를 사용한다.

![this 키워드 사용](/assets/20210624_002101.png)

위의 기존 함수를 인스턴스 메서드로 만들고 싶을 때, `this` 키워드로 객체를 넘겨받고 `static`을 빼면 위와 같이 된다. 즉, `static`과 파라미터를 지우고 `this`만으로 인스턴스 메서드를 사용하는 함수로 바꾼 것이다.

## HAS-A 관계 (구성/집합)

일반적으로는 사용 관계와 구성 관계를 갖도록 이렇게 사용한다.

- **composition(구성):** 일체형이라고 생각하면 된다. 부품에 해당하는 객체가 1:N이든 1:1이든, 그것을 소유한 객체와 동일한 시점에 생성되고 동일한 시점에 사라지는 관계를 말한다.
- **aggregation(집합):** 1:1이든 1:N이든 상관없지만 동일한 생명주기를 가지지 않는다는 점이 차이다. 일반적으로 컬렉션으로 수집되는 관계를 말하며, 부품을 소유한 객체가 만들어져도 부품 객체는 없을 수 있고, 반대로 부품을 소유한 객체가 사라져도 부품 객체는 별도로 생존할 수 있는 관계를 말한다.
