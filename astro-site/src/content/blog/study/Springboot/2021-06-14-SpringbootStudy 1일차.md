## 1일차 (2021-06-14)

브라우저에서 요청을 보낸다는 것은 url 경로를 주소창에 입력한다는 뜻이다. 그러면 (예를 들어) 네이버 서버에 요청을 보내게 되고, 요청이 왔으니 네이버 메인 페이지를 보여준다.

요청이 오면 서버에서 응답해주는데, 그 일을 컨트롤러가 한다. 컨트롤러는 어떤 요청에 어떤 메서드가 매핑되는지를 정한다. 그 메서드 안에서 글쓰기, 글 조회, 목록 조회 같은 처리가 들어온다.

예) `localhost:8080/board/list` 요청 시 view 페이지를 보여줘야 한다.

컨트롤러에서 하는 작업은 다음과 같다.

1. **요청 경로 매핑**: 요청과 어떤 메서드(컨트롤러의)를 연결한다.
2. **서비스 호출**: 게시글 목록을 DB에서 조회한다.
3. 데이터가 컨트롤러까지 오면 모델에 담는다.
4. 게시글 목록을 모델에 담아두고, 어떤 뷰 페이지에 담을지 뷰를 선택한다.

모델에 속성을 추가하면, 뷰(jsp나 타임리프 html 등)에서 꺼내와 쓴다. 서비스에 추상 메서드를 두고 어떤 DAO·매퍼로 가는지 정해주며, 최종적으로 html로 응답을 보낸다.

- Spring Starter Project
- war, jar는 배포·호스팅할 때 편하다. 지금은 jar로 한다(나중엔 war).

### 제어의 역전 (IoC / DI)

제어의 권한은 원래 개발자가 갖고 있다. 객체가 언제 소멸하고 생성되고 의존하는지를 다 개발자가 만든다. 그런데 그것들을 하나하나 만들면 정작 핵심 로직에 투자할 시간이 줄어든다. 그래서 객체 생성 같은 것을 전부 프레임워크에 넘기는 것이다.

객체의 생명주기를 프레임워크에 맡기고 핵심 로직에 집중한다. 그 개념 아래에 의존성 주입(DI)이 있다. 객체가 `new`로 무한 생성되지 않고, 스프링 컨테이너 하나에 객체 하나가 등록되면 그 컨테이너에서 가져다 쓸 수 있다.

우리가 해줄 일은 변수에 `@Autowired`를 붙여주는 것이다. 그러면 의존성 주입이 실현되어, 위에서 추상 메서드로 정의해둔 것을 자동으로 가져온다. (암호화 같은 작업을 하고 매퍼로 넘긴다.)

### 컨트롤러 → 서비스 → 매퍼 → html

- 매퍼: 네임스페이스는 BoardMapper 인터페이스에 대한 경로이며, 이게 xml과 연결되어 있다고 지정한다.
- BoardController에서 서비스에 register를 넘겨준다.

화면에서 서버로 넘어오면 서비스로 요청을 보내고, register가 실행되며, 구현 클래스의 register가 동작한다. 그것이 매퍼의 insert 메서드를 호출하면, 매퍼 xml에서 insert라는 id를 가진 쿼리가 실행된다.

id가 `insert`인데, insert id와 BoardMapper 인터페이스를 일치시켜주면 된다.

**연결 흐름**: BoardController에서 register 호출 → 서비스의 register 실행 → Impl의 insert 실행 → BoardMapper의 insert 실행 시 매핑된 xml 쿼리 실행. 이때 board 객체에 담긴 파라미터를 가져온다.

`utf8mb4` 인코딩으로 board 테이블을 만든다.

![board 테이블 생성](/assets/20210614_231430.png)

**DAO / DTO / VO**

- 원래는 DAO에서 pstmt로 요청한다.
- DTO는 DB로 전송할 때 그 사이에서 데이터를 감싸는 객체이다. title, content 같은 변수를 board라는 객체 하나로 묶으면, DB에서 조회되는 데이터 한 건이 board 하나로 묶인다. 그걸 리스트로 받아온다.
- DTO는 데이터를 전송하기 위해, 데이터베이스의 컬럼과 자바 객체를 매핑한다. (테이블에 저장한 걸 그대로 가져온다.)
- DAO는 여러 컬럼이 담긴 바구니를 그대로 가져오는 것인데, 그걸 매퍼(Mapper)가 대신한다. `?, ?, ?`를 매퍼가 대신해주고(MyBatis 프레임워크가 대신), 추가로 쿼리와 자바 코드를 분리해준다.

## 2일차 (2021-06-18)

DB에서 조회한 목록을 보여준다.

BoardService에 list를 만들었다. 게시글 목록을 요청했을 때 `list.html`에서 모델에 담긴 데이터를 뿌려줘야 한다.

- BoardController에서 "list"라는 이름으로 모델에 등록한다.
- list라는 이름의 목록을 가져와 표현식으로 뿌려준다.
- 각 리스트 요소를 board 객체로 가져온다. (반환 결과가 `List<Board>` 컬렉션이므로)
- board라는 이름으로 `<tr>` 태그를 반복 출력한다.

게시글 목록은 데이터 한 건이 아니라 여러 글이므로 컬렉션 리스트로 받아온다. 반면 게시글 조회는 `Integer boardNo`로 한 건만 조회하므로 Board 객체 하나만 가져오면 된다. boardNo를 넘겨주면서 해당 글번호의 게시글 정보를 조회해 객체로 받아와 모델에 등록한다.

**BoardController**

```
package aloha.controller;

import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.stereotype.Controller;
import org.springframework.ui.Model;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.PostMapping;
import org.springframework.web.bind.annotation.RequestMapping;

import aloha.domain.Board;
import aloha.service.BoardService;

@Controller
@RequestMapping("/board")
//경로
public class BoardController {

	@Autowired
	//의존성 자동주입
	private BoardService service;

	//게시글 쓰기 - 화면,처리

	@GetMapping("/register")
//	@RequestMapping(vaule="/register",method = RequestMethod.GET)
	public void registerForm(Model model, Board board) throws Exception{

	}

	@PostMapping("/register")
	public String register(Model model, Board board) throws Exception{
		//글쓰기 요청
		service.register(board);	//DTO 넘김

		model.addAttribute("msg", "등록완료");
		return "board/success";	//이동

	}

	//게시글 목록
	@GetMapping("/list")
	public void list(Model model, Board board) throws Exception{

		model.addAttribute("list",service.list());
		//전달받은거 모델등록
	}

	//게시글 읽기

	@GetMapping("/read")
	public void read(Model model, Integer boardNo) throws Exception{

		model.addAttribute("board",service.read(boardNo));

	}

	//게시글 수정화면

	@GetMapping("/modify")
	public void modifyForm(Model model, Integer boardNo) throws Exception{

		model.addAttribute("board",service.read(boardNo));
	}


	//게시글 수정처리

	@PostMapping("/modify")
	public String modify(Model model, Board board) throws Exception{

		service.modify(board);	//board객체 받아옴
		model.addAttribute("msg","수정 완료되었습니다.");
		return "board/success";

	}

	//게시글 삭제처리

	@PostMapping("/remove")
	public String remove(Model model, Integer boardNo) throws Exception{

		service.remove(boardNo);
		model.addAttribute("msg","삭제 완료되었습니다.");
		return "board/success";

	}


	//성공
}


```

board 경로에 해당하는 뷰 페이지 정보를 화면에 출력한다. 예를 들어 `boardNo=2` 경로로 요청하면, 글 읽기 요청을 클라이언트로 응답해주면 된다.

**작업 순서**

1. BoardController 등록
2. 매퍼 인터페이스 등록
3. 서비스 등록 (서비스도 인터페이스)
4. 서비스 Impl 구현 (인터페이스 구현)
5. 매퍼 파일 구현
6. BoardMapper에 쿼리 추가
7. 다시 BoardController 등록

**타임리프 `th` 사용**

화면에 데이터를 뿌릴 때, 타임리프 템플릿 엔진에서 `th:` 콜론으로 값을 넣는다. board 객체를 board라는 이름으로 담았으므로, 객체인 경우 상위 태그에 등록해두고 폼 태그의 하위 태그(내부 요소)에서 변수들만 접근해 쓴다. `th:value`나 `th:field`로 쓰며, 애스터리스크(`*`)를 사용해 객체를 가져오면 뒤에 변수만 써서 작성할 수 있다.

수정 버튼을 눌러 수정 화면으로 가게 하려면, 컨트롤러에서 RequestMapping 요청을 받아 수정 화면을 응답하는 게시글을 만든다. 절대 경로로 board의 remove로 요청한다.

**Spring Model 객체**

Controller의 메서드는 `Model` 타입의 객체를 파라미터로 받을 수 있다. 순수 JSP/Servlet으로 웹 애플리케이션을 만들 때는 보통 request나 session 내장 객체에 정보를 담아 jsp에 넘겨주곤 했는데, Spring에서는 `Model`을 쓴다.

- GET, POST 둘 다 한 경로로 사용 가능하다.
- 로그인 정책, 조회수, 파일 업로드/다운로드 등 다양하게 가능하다.
- 게시글이 많으면 페이징 처리도 해야 한다.

**매퍼와 서비스를 나눈 이유**: 서비스에서 로직을 더 추가하기 위함이다. 구현체에서 비즈니스 로직을 구현한다. 기능 하나(글쓰기, 회원가입 등)가 실행될 때 필요한 모델과 일련의 과정들을 비즈니스 과정이라 한다. (예: 회원가입할 때 비밀번호를 암호화해서 넣는다.)

## 3일차 (2021-06-21)

검색 기능을 구현한다.

![검색 구현 흐름](/assets/KakaoTalk_20210621_194801492.png)

컨트롤러에 내용을 넣고 모델에 담아 서비스로 보내고, 매퍼를 통해 DB로 간 다음, 받은 내용을 모델에 담아 컨트롤러로 가져오고, 그 내용을 뷰로 보여준다.

```
use aloha;
select *
from board
-- where title like concat('%', #{keyword},'%')
where title like concat('%', #{keyword},'%')
	or content like concat('%', #{keyword},'%')
    or writer like concat('%', #{keyword},'%');


select *
from board
where reg_date >= date(now())-30;


select *
from board
where reg_date >= date(subdate(now(), interval 7 day));

-- 1 페이지
select *
from board
limit 0,10; -- index(순서번호), count(개수)

-- 2 페이지
select *
from board
limit 10,10; -- index(순서번호), count(개수)

-- 3 페이지
select *
from board
limit 20,10; -- index(순서번호), count(개수)


-- 일반적인 페이지
select *
from board
limit #{startRowIndex}, #{rowsPerPage};




```

### 페이징 처리

**필수 값**

- 현재 페이지 번호: `pageNum`
- 페이지당 게시물 수: `rowsPerPage`
- 전체 데이터 개수: `totalCount`
- 노출 페이지 개수: `pageCount`

**수식**

- 시작 페이지 번호(`startPage`): `((현재 페이지 번호 - 1) / 노출 페이지 개수) * 노출 페이지 개수 + 1`
- 끝 페이지 번호(`endPage`): `((현재 페이지 번호 - 1) / 노출 페이지 개수) + 1) * 노출 페이지 개수`
- 첫 번째 페이지 번호(`firstPage`): `1`
- 마지막 페이지 번호(`lastPage`):
  - `totalCount % rowsPerPage == 0`이면 `(전체 데이터 개수 / 페이지당 게시물 수)`
  - 아니면 `(전체 데이터 개수 / 페이지당 게시물 수) + 1`
- 이전 페이지 번호(`prev`): `현재 페이지 번호 - 1`
- 다음 페이지 번호(`next`): `현재 페이지 번호 + 1`
- 시작 글 index, 검색어(`keyword`)

**예시**

전체 데이터 개수 127, 페이지당 게시물 수 10, 노출 페이지 개수 10, 현재 페이지 번호 15일 때:

- 시작 페이지 번호: 11
- 끝 페이지 번호: 20 → 13
- 첫 번째 페이지 번호: 1
- 마지막 페이지 번호: 13
- 이전 페이지 번호: 14
- 다음 페이지 번호: 16

**계산 검증 (총 212건, 10건씩, 22페이지, 현재 14페이지)**

시작 페이지:

```
(11-1 / 10) * 10 + 1 ---> 11
(14-1 / 10) * 10 + 1 ---> 11
15 ---> 11
16 ---> 11
17 ---> 11
.
.
(20-1 / 10) * 10 + 1 --> 11
```

끝 페이지:

```
((11-1 / 10) + 1) * 10 --> 20
..
((20-1 / 10) + 1) * 10 --> 20
```

노출 페이지 개수가 15일 때 시작 페이지:

```
((1-1)/15) * 15 + 1 --> 1
((14-1)/15) * 15 + 1 --> 1
...
((15-1)/15) * 15 + 1 --> 1
```

현재 페이지 33일 때:

```
시작: ((33-1)/15) * 15 + 1 --> 31
끝:   ((33-1)/15 + 1) * 15 --> 45
```

구간 예시: `1~15`, `16~30`, `31~45`, `46~60`

정리하면 (현재 14페이지 기준):

- 시작: 11, 끝: 20
- 첫: 1, 마지막: 22
- 이전: 13, 다음: 15

## 7일차 (2021-07-05)

`service.read`로 모델에 데이터를 담는다. 모델에 담은 것은 뷰 페이지에서 쓸 수 있다. `files`라는 모델 속성을 가져와 `th:each`로 file 객체를 반복한다.

**파일 다운로드 요청 처리**

`fileDownload`로 요청을 보내면, 그 요청을 처리할 컨트롤러를 만들고 URI 매핑을 해줘야 한다. `@RequestParam`을 `Map` 컬렉션으로 전달받겠다.

**버퍼를 이용한 전송**

파일을 생성·복사할 때, 파일은 용량(바이트)이 있는데 컴퓨터가 한 번에 몇 기가·몇 테라를 입출력으로 전송해줄 수는 없다. 시스템 안에서 다른 공간에도 메모리가 할당되어 있어, 하나의 프로그램 안에서 몇 테라씩 주고받기는 힘들다.

그래서 어느 정도 단위로 버퍼에 실어, 그 버퍼씩 데이터를 보내고 전달받아 다시 파일로 보내주는 작업이 필요하다.

**파일 다운로드의 의미**: 이 컴퓨터에 저장된 파일을 외부로 전송하는 것이다. 즉 업로드되어 있는 파일을 읽어들여 웹 페이지(웹 애플리케이션)로 전송함으로써 다운로드시킨다. 파일 객체를 가져와 정상적으로 전송하기 위한 작업이 필요하다. (브라우저마다 인코딩 방식이 달라 파일 데이터·파일 이름을 그에 맞게 넘겨줘야 한다.)

`read.html`에서 파일 경로로 `fullname`과 `filename`을 요청 보내면, `fileDownload`로 매핑된 곳에서 전체 경로와 파일명을 받아 `InputStream`, `OutputStream`으로 서버로 전송하는 작업을 한다.

---

**스트림 객체**

- 파일을 PC에서 가져오려면 `FileInputStream` 객체가 필요하다.
- 반대로 파일을 전송하는 쪽(웹 페이지)에는 `ServletInputStream`, `ServletOutputStream`이 있는데, 출력을 웹 페이지로 해준다.

가장 먼저 해줄 작업은 브라우저에 맞는 인코딩 방식을 적용하는 것이다. 파일 데이터 자체는 원본을 가져오지만, 파일명은 브라우저(엣지, IE, Chrome 등)마다 적용 방식이 다르다. 브라우저에 정상적으로 인코딩하려면 그에 맞게 인코딩해야 한다.

그다음 response 객체에 헤더 정보를 붙인다. (웹 페이지로 응답하기 위한 규칙을 만든다.) 전체 경로로 가져온 파일 객체로 InputStream을 만들고, `ServletOutputStream`은 응답 객체에서 가져온다. `getOutputStream`을 호출하면 어떤 형태로 출력 스트림을 만들지 그 인스턴스를 메서드를 통해 가져올 수 있다.

> **수도관 비유**: 강에서 아파트까지 연결된 굵은 관이 `FileInputStream`이고, 아파트에서 우리 집 수도꼭지까지 오는 관이 `ServletOutputStream`이다. 강에서 끌어오는 관은 물을 많이 가져오려고 굵고 튼튼해야 하고, 집까지 오는 관은 손 씻기에 적합한 관이다. (두 관은 당연히 다르게 생겼다.)

---

서버 컴퓨터의 파일을 가져오는 연결 통로는 `FileInputStream`이고, 웹 페이지에 보여주는 것은 `ServletOutputStream`이다. 그 인스턴스는 `response.getOutputStream()`을 호출해 가져올 수 있다.

---

**`FileInputStream`은 `new`인데 `OutputStream`은 왜 `new`가 아닌가?**

`OutputStream`은 response 객체가 가진 인스턴스를 가져오는 것이다(`response.getOutputStream()`).

> **연결 통로 비유**: 우리 집으로 들어오는 관에는 수도관, 랜선, 가스관 등 여러 개가 있다. 다만 우리 집으로 들어오는 (수도)관은 하나로 만들고, 화장실1, 화장실2, 싱크대 등으로 나눠 쓴다. 마찬가지로 `OutputStream`도 `new`로 여러 개 만드는 것보다 하나로 만드는 게 낫다. `new`는 객체를 생성할 때마다 서버 성능이 저하되기 때문이다.

**버퍼 읽기/쓰기**

입력해서 가져온 파일을 조금씩 출력한다. 바이트 배열을 만들어 1024바이트(=1KB)를 담는다. 파일이 1MB면 1KB씩 1024번 가져와야 하는데, 이를 한 번에 가져오는 게 아니라 `FileInputStream`의 `read`로 1KB씩 조금씩 읽어온다. 더 이상 가져올 데이터가 없으면 `-1`을 반환하므로, `-1`을 만나면 반복을 끝낸다.

가져온 데이터를 `ServletOutputStream`으로 1KB씩 출력한다. 버퍼에 담아 웹 페이지로 조금씩 보내주는 것이며, 다운로드 시 조금씩 쌓여 한 파일이 된다.

---

한글명으로 된 파일(예: "강아지")을 특정 브라우저에서 되도록 인코딩하고, 헤더에 설정을 적용한 뒤, input/output 스트림으로 조금씩 내려받을 수 있도록 `OutputStream`으로 출력한다.

---

**조금씩 가져오는 이유**

서버는 한 번에 전송할 수도 있지만, 그러기 어려운 이유가 있다.

> **택배 비유**: 청바지 1장은 문제없지만, 2천 장을 한 번에 배송하라 하면 큰 트럭이 필요하고 우리 집에 주차할 방법이 없다.

마찬가지로 4GB 영화를 받으려는데 내 컴퓨터에 4GB를 한 번에 처리할 만큼의 메모리(RAM) 공간이 확보되어 있지 않으면(예: 8GB 중 6GB 사용 중) 한 번에 다운받기 힘들다. 10MB씩 받으면 메모리 할당량도 적고 데이터 유실 가능성도 적어진다.

### 받아들일 수 있는 메모리 용량의 한계, 메모리 유실 가능성

데이터를 요청했는데 문제가 있으면 다시 전달받게 하는 프로토콜 방법도 있다. 이런 여러 이유로 조금씩 가져온다. (100GB 게임을 한 번에 못 받는 것과 같다. 컴퓨터 성능뿐 아니라 인터넷 회선 성능의 한계도 있다.) 그래서 통신·네트워크 지식도 필요하다.

> 참고: `Ctrl+H`로 파일을 검색해 찾을 수 있다.

**전체 다운로드 (압축)**

파일이 여러 개일 때는 게시글 이름으로 압축 파일 이름을 지어주도록 한다. 게시글 이름과 번호를 넘겨주면, 게시글 번호로 딸려 있는 파일들을 조회해 전체 경로를 찾아올 수 있다. 예외가 발생할 수 있으니 `exception`을 추가한다. (한 번에 import하려면 `Ctrl+Shift+O`)

파일 리스트의 파일들을 하나씩 꺼내, `fullname`(전체 경로)으로 파일을 `FileInputStream`으로 읽고, 조금씩 읽어 zip 파일에 출력한 뒤 최종적으로 응답한다.

- 단일 파일 때는 `ServletOutputStream`이었지만, 지금은 zip 파일(`ZipEntry`)이다.
- 파일 하나씩을 `InputStream`으로 읽어와 `ZipOutputStream`으로 압축 파일을 만들고, 그 압축 파일을 `ServletOutputStream`으로 웹 페이지에 출력한다.
- `ZipEntry`는 압축 파일에 들어가는 항목 하나하나를 말한다. 파일 이름으로 파일 객체를 만들고, 그 이름 기준으로 항목을 만들어 다음 엔트리에 넣는다.
- 실제 데이터는 `zout.write(buffer, 0, length)`에서 넣어준다.

파라미터를 넘길 때 `&`로 연결하며, 그 뒤의 객체에는 `$`가 없어도 된다.

**파일 삭제**

파일명 뒤의 x 번호를 누르면 파일이 삭제되게 한다. (x 아이콘이 필요하다.) Ajax로 어떤 경로·방식(GET/POST)으로 요청 보낼지, 어떤 파라미터를 넘길지 지정한다. 우리는 `fileNo`와 `filePath`를 넘긴다. 그리고 이 span 태그의 부모 요소를 delete해서 없애준다.

### 비동기 통신

Ajax는 비동기로 통신한다.

글쓰기는 동기식이다. 제목·내용을 쓰고 완료한 뒤, 다시 글쓰기 화면을 요청해야 두 번째 글을 작성할 수 있다. 그런데 지금처럼 한 화면 안에서 파일을 삭제하고, 처리가 끝나면 화면이 그대로 유지되어야 하는 경우가 있다.

- **동기식**: form 태그로 동작하며, 화면이 재응답(한 번 새로고침)된다.
- **비동기식**: 화면이 그대로 유지된다.

파일 삭제나 댓글 같은 것은 비동기식으로 해야 한다. 만약 댓글 쓰기 화면이 따로 열리고 다시 게시글로 이동하면 매우 불편할 것이다. 댓글은 게시글 상태에서 비동기로 쓰고, 그 댓글 부분만 새로 응답한다. (내가 쓴 댓글이 있는 목록만 다시 뿌려주면 된다.)

![비동기 댓글 처리](/assets/20210705_203143_8cuu6wzgd.png)

그러면 페이지 전체가 다시 불러와질 필요 없이, 파일만 삭제되고 전체는 유지된다.

**RESTful API**

조회·수정·삭제 등을 정확히 지키면서 작성하는 것이 RESTful이다. 풀스택이면 GET/POST만 써도 되지만, 프론트와 백을 나누면 RESTful로 해야 한다.

- HTTP URL을 통해 자원을 명시하고, HTTP 메서드(GET, POST, PUT, DELETE)로 해당 CRUD를 적용한다.
- 조회 요청이면 어디로, 삭제 요청이면 어디로 가게 하는 등 규칙에 맞게 매핑한다.
- 이런 게 필요해진 이유는 클라이언트가 다양화되었기 때문이다. 서버는 규칙대로 만들고, 클라이언트도 규칙대로 작업한다.

이것이 REST의 기본 틀이다. (꼭 이래야 하는 건 아니지만 서로 간의 약속이 맞아진다.) 클라이언트의 요청이 서버에서 통용되는 방식으로 적용되어야, 수정 요청한 것이 수정 방식으로 사용된다. 파일 확장자를 mapping에 안 쓰거나 언더바를 안 쓰는 등의 규칙도 있다.

---

작성된 파일 목록을 비우고, 남아 있는 파일 목록을 갱신하는 방법으로 사용할 수도 있다. board 패키지 밑의 success 파일을 매개변수로 응답시켜, 데이터를 `board/success`로 보낸다.

**(개발자의 공부에 대하여)**

개발자로 취업해도 공부는 계속해야 한다. 어떤 기술을 한 번에 딱 이해하면 좋겠지만 그렇게는 안 된다. 한 번 학습하고 복습을 100번 해야 한다. 과정은 `학습 → 검색 → 복습 → 응용`이다.

검색도 시간이 든다(책, 구글링, 이전 코드, 질문 등). 프로젝트에 적용하려면 그때 공부한 게 기억 안 나서 또 검색하고 복습하고 응용하게 된다. 회사에서도 한 번 하고 끝이 아니라 또 찾아보고 응용하는 식이다.

결국 검색을 가장 많이 하게 된다. 공부할 수 있는 시간이 300시간이라면 검색 시간이 길어질수록 시간을 뺏긴다. 그래서 검색 시간을 줄여야 하는데, 가장 좋은 것은 나만의 책(블로그)을 만드는 것이다.

블로그에 배운 것을 카테고리·주제로 작성해두면 검색 과정이 단순화된다(시간 절약). 이렇게 컨텐츠가 쌓이면 어마어마해지고, 블로그 + 깃허브가 프로젝트 히스토리가 된다.

**외래키와 파일 삭제**

- xml은 서버를 켜서 확인한다.
- 글을 삭제하면 파일이 남아 있으면 안 되고, 전체 다운로드도 없어져야 한다.

board와 board_attach는 boardNo로 연관관계를 맺는다. board_attach를 외래키로 지정해 board 테이블의 기본키(board_no)를 참조하게 만든다. board 테이블의 게시글 하나가 사라지면, 외래키로 저장해 둔 attach의 번호도 사라져야 한다. board 테이블의 board_no를 지우면 board_attach가 지워지게 하는 것이 `ON DELETE CASCADE` 방식이다.

외래키는 DB에 대한 작업이고, BoardController에서는 실제 파일도 삭제해줘야 한다. 파일 리스트를 조회해와 반복하면서 삭제한다.

## 8일차 (2021-07-11)

**댓글 처리**

댓글을 작성하는 것은 글 읽기 화면에서 해당 게시글 밑에 작성하는 것이다. 먼저 댓글 목록을 조회한다.

**댓글 목록 조회**

- `board/list`가 아니라 다른 방식으로 요청한다.
- 댓글은 ① 게시글 읽기 화면에서 보이고, ② 등록할 때도 보이고, ③ 수정할 때도 변경되어야 한다.

서비스 요청은 `public List<Reply> replyList(Integer boardNo);`로, 글번호를 넘겨 여러 댓글을 가져온다.

### Mapper

쿼리는 reply 테이블에서 조회하게 한다. 보여질 뷰 페이지는 `reply/list.html`이다.

**댓글 글쓰기**

- `board/replyRegister`로 POST 요청만 보낸다. (따로 화면 없을 예정)
- 댓글 자체로만 매핑되어야 한다. (GET으로 매핑되는 부분이 없고 POST로만 보낼 예정)
- 서비스에서 `replyRegister`로 객체를 넘겨 등록한다. (`void replyCreate(Reply reply)`)
- html은 따로 존재할 것이 없다. 댓글 등록 후 `reply/list.html`이 응답될 것이다.

---

댓글 목록을 새로 조회하고 그 화면을 새로 보여준다. 제이쿼리에서 Ajax로 비동기 요청을 보낸다. 그 전에 수정부터 만들자.

**댓글 수정**

- 댓글 등록과 유사하다. (`board/replyModify`로 댓글 수정 처리)
- 게시글 읽기 페이지 자체에서 수정 영역이 나오고, 그 영역에서 수정하므로 따로 request 요청이 없다.
- 매퍼에서는 insert 대신 update로 작업한다.
- 변경해야 하는 것은 content이다. (writer는 실제로는 수정하지 않는다.) 조건은 해당 게시글이며, 댓글 번호를 기준으로 수정한다.

댓글을 수정하고, 수정된 내용까지 포함된 댓글 목록을 다시 응답받아 글 읽기 화면에서 그 영역만 세팅한다. 새로 등록된 댓글을 포함한 댓글을 다시 가져오고, 그 화면을 "데이터"라는 이름으로 뷰 페이지에서 가져온다. 기존 댓글 목록은 `empty()`로 클리어하고, `append()`로 새 엘리먼트를 추가한다.

- 댓글을 작성했으면 댓글 입력 상자를 비워준다.
- `writeReply` 함수: boardNo를 가져와 호출하면, Ajax로 비동기 요청을 보내 BoardController에 매핑된 객체를 Reply로 받아 `replyRegister`로 등록하고, 글번호로 댓글 목록을 가져와 모델에 등록해 `reply/list.html` 뷰 페이지를 응답한다.

응답하면 `read.html` 안의 요청에서 "데이터"라는 매개변수 이름으로 `reply/list.html` 뷰 페이지를 전달받고, 그 안에서 갱신된 댓글 내용이 넘어온다. 기존 댓글 목록은 비우고 새로 갱신한다.

댓글 이벤트를 다시 가져오면 다시 실행해줘야 하는 의무가 있다. (수정·삭제 요청 처리)

## 9일차 (2021-07-14)

**댓글 수정 시 에디터 처리**

- 댓글 수정 시 사라지게 하는 게 아니라, 다른 댓글의 에디터는 다 없애고 현재 것만 추가한다.
- 취소를 누르면 에디터가 비워지게 한다.

이 이벤트 안에서 에디터 태그가 만들어진다. 그 버튼에 대한 등록 이벤트·취소 이벤트를 만들어야 하는데, 댓글 수정 클릭 이벤트 안에서 정의해야 한다. 수정 버튼을 클릭했을 때 등록·취소 버튼이 나와야 하고, 이 둘은 수정 버튼을 클릭해야만 뜨기 때문이다. 그래서 이 내부에서 버튼에 대한 클릭 이벤트도 내부에 정의해야 한다.

`replyNo`만 알아도 조회 가능하지만, 게시글을 갱신해도 조회 가능해야 해서 다시 목록을 조회하기 위해 boardNo를 넘긴다.

---

수정 버튼이 삽입될 때 등록 버튼이 에디터 태그를 만들어 현재 화면에 붙여준다. 이 등록 버튼은 수정 버튼을 클릭한 시점에 발생할 이벤트이므로, 수정 버튼이 생긴 그 시점에 클릭 이벤트를 정의해야 한다. (수정 버튼 바깥에 존재하면 동작하지 않는다. 안에 넣어줘야 정의한 함수가 실행 가능하다.)

**목록 갱신 충돌 문제**

댓글 목록이 갱신되면 `reply/list.html`을 다시 가져온다. list도 갱신되어야 해서 `list.html`도 바꿔줘야 한다. 그런데 새로 가져온 `list.html`에는 댓글 삭제·수정·수정 버튼 클릭에 대한 이벤트가 없어서, 자바스크립트 웹 문서를 2번이나 가져오게 되어 충돌이 생긴다.

근본 원인은 글 읽기(`board/read?boardNo=10`)에서 다음 두 가지 로직이 겹친다는 점이다.

- 댓글 목록 조회 → 댓글 등록/수정 → 댓글 목록 갱신
- (별도) 댓글 목록 조회

**해결: Document Ready 이벤트 사용**

그러면 애초에 댓글 목록 조회를, 서버에서 하지 말고 화면이 열린 뒤에 하면 되지 않을까?

`function`을 `$(document).ready()`(the Document Ready Event) 함수로 감싸면, 모든 문서가 준비되었을 때 실행된다. 즉 문서가 떴을 때 댓글 목록 조회를 요청한다는 뜻이다.

댓글 목록이 조회되어 들어올 때 `reply/list.html`을 불러오고, reply와 관련된 함수는 `reply.js`에 넣는다.
